# 나만의 지식/노하우 위키 — 운영 규칙 (Schema)

이 저장소는 **삼대오백 / MXSSIVE 브랜드 운영자의 개인 나무위키**다.
Claude는 이 파일의 규칙에 따라 위키를 작성·관리한다. 사용자는 자료를 넣고 질문만 한다.

---

## 0. 비즈니스 맥락 (모든 작업의 기준)

> ⚠️ 모든 위키 페이지의 takeaway는 아래 맥락을 기준으로 작성한다.
> 자세한 내용은 [[00-비즈니스-맥락]] 페이지 참조.

- **운영자 역할**: 영업 · 마케팅 · 구매 관리자 (유통)
- **기존 브랜드**
  - **삼대오백** — 헬스식품 / 건강식품 (전문 헬스인 타겟)
  - **MXSSIVE** — 헬스장비
- **준비 중인 신규 라인**: **WELLNESS**
  - 삼대오백 기반이되 **일상 · 라이트 · 루틴** 지향
  - **여성향(여성 타겟) 제품**
  - 전문 헬스인보다 넓은 일반 소비자층

신규 자료를 정리할 때는 항상 **"이게 삼대오백/MXSSIVE 운영과 WELLNESS 신규 라인에 어떤 의미인가?"**를 먼저 묻는다.

## 1. 폴더 구조

```
/
├── CLAUDE.md          ← 이 파일. 운영 규칙
├── raw/               ← 원본 자료 (사용자가 넣음, Claude는 읽기만)
│   └── README.md
├── wiki/              ← 정리된 지식 (Claude가 작성·관리)
│   ├── index.md       ← 전체 목차 (카테고리별 분류)
│   ├── log.md         ← 작업 이력
│   ├── 00-비즈니스-맥락.md
│   └── *.md           ← 각 지식 페이지
└── 모니터링/           ← 브랜드 일별 모니터링 로그 (시계열 데이터)
    ├── 00-모니터링-목차.md
    └── YYYY-MM-DD.md  ← 날짜별 스냅샷
```

- **`raw/`와 `wiki/`는 하위 폴더 없이 플랫하게 관리**한다. 분류는 폴더가 아니라 **`wiki/index.md`의 카테고리**로 한다.
- **`모니터링/`은 예외**: 브랜드 일별 모니터링 로그 전용 폴더(시계열 데이터). 내부는 `YYYY-MM-DD.md`로 플랫하게 쌓는다.
  - 이 일별 로그는 **데이터 기록**이므로 §3의 takeaway callout/표준구조 의무에서 제외하고, 대신 맨 위 `> 관전포인트:` 한 줄만 둔다.
  - 인사이트 종합은 `wiki/`의 월간 페이지(`브랜드-모니터링-YYYY-MM`)에서 하며, 이쪽은 §3 규칙을 따른다.
- Claude는 `raw/`를 **절대 수정하지 않는다** (읽기 전용 입력).

## 2. 운영 방법 — 3가지 명령

### ① 자료 넣기 (Ingest)

사용자가 `raw/`에 파일을 넣거나 내용을 붙여넣고 "정리해줘"라고 하면:

1. 원본을 `raw/`에 보관한다. (사용자가 직접 넣었으면 그대로 둠. 채팅으로 준 내용이면 `raw/`에 파일로 저장)
2. 내용을 읽고 **핵심을 추출**한다.
3. `wiki/`에 마크다운 페이지를 작성/갱신한다. 작성 규칙은 **§3** 참조.
4. **관련된 기존 페이지와 `[[위키링크]]`로 교차참조**를 건다. (양방향으로 — 새 페이지 → 기존 페이지, 기존 페이지 → 새 페이지)
5. `wiki/index.md`의 해당 카테고리에 항목을 추가/갱신한다.
6. `wiki/log.md`에 한 줄 기록한다. (날짜, 작업, 영향받은 페이지)

> 한 자료가 여러 주제를 담으면, 주제별로 페이지를 나누는 것을 우선한다. 페이지는 작고 단일 주제로.

#### 정기 공급원: 뉴스·블로그 검색 루틴

`raw/`에 `*_뉴스루틴_*.txt` 형태의 파일이 들어오면, 이는 **네이버 API 기반 모니터링 루틴**의 산출물이다. (상세: [[뉴스검색-루틴]])

> **운영 모드: 반자동.** 수집은 평일 10시 작업 스케줄러가 자동 실행(`scripts/fetch-news.ps1`)하여 `raw/`에 저장한다. **정리는 사용자가 `/뉴스정리` 명령**(또는 "뉴스루틴 정리해줘")으로 트리거한다.

`/뉴스정리` 또는 "뉴스루틴 자료 정리해줘" 요청 시:

1. **브랜드 일별 노출** → `모니터링/YYYY-MM-DD.md` 신규 생성 (자사+경쟁사). 추적 대상: **삼대오백(자사), 익스트림, 잠백이, 이그니스, 제로투히어로**. `모니터링/00-모니터링-목차.md` 상단에 링크 추가.
2. 신규 **시장·트렌드** → `wiki/단백질-건기식-시장동향-YYYY-MM` 페이지 갱신(같은 달) 또는 신설(새 달, 이전 달과 교차링크).
3. **브랜드 움직임 종합** → `wiki/브랜드-모니터링-YYYY-MM` 페이지 갱신/신설 (일별 로그를 월간 인사이트로 압축).
4. **규제·법령** 이슈 → [[건기식-광고규제-과징금강화]] 등 제품·법령 페이지 갱신.
5. 네이버 API 키 등 **시크릿은 위키 본문에 절대 적지 않는다** (파일 위치만 참조).

### ② 질문하기 (Ask)

사용자가 질문하면:

1. `wiki/index.md`로 관련 페이지를 찾는다.
2. 해당 위키 페이지들을 읽고 **근거 기반으로 답한다**. (출처 페이지를 `[[링크]]`로 명시)
3. 위키에 없는 내용이면 "위키에 없음"을 분명히 밝히고, 일반 지식으로 답할 땐 그렇다고 표시한다.
4. 답하면서 발견한 **새로운 인사이트·빈틈**은 사용자에게 "위키에 추가할까요?"로 제안한다.

### ③ 건강검진 (Health Check)

사용자가 "건강검진" / "점검해줘"라고 하면 위키 전체를 진단한다:

- **끊긴 링크** — `[[링크]]` 대상 페이지가 없는 곳
- **고아 페이지** — index에 없거나 아무도 링크하지 않는 페이지
- **중복/분할 후보** — 내용이 겹치거나 너무 커진 페이지
- **index 불일치** — index와 실제 파일 목록의 차이
- **오래된 정보** — 갱신이 오래되어 재검토가 필요한 페이지 (특히 시장 동향·법령)
- **takeaway 누락** — §3 규칙(맨 앞 callout)을 안 지킨 페이지

결과를 리포트로 보여주고, 고칠 항목을 제안한다. 승인 시 수정하고 `log.md`에 기록한다.

## 3. 위키 페이지 작성 규칙 (가장 중요)

### 3-1. 맨 앞에 반드시 takeaway callout

모든 페이지는 **제목 바로 아래에 `>` callout 블록**으로 시작한다.
이 callout은 **내 비즈니스/방향성 관점의 핵심 takeaway**다 — "이 내용에서 내가 가져가야 할 게 뭔가"를 첫 문단에서 바로 알 수 있게.

```markdown
# 페이지 제목

> [!takeaway] 내 관점 핵심
> - (삼대오백/MXSSIVE/WELLNESS 관점에서 이 내용의 의미 한두 줄)
> - (당장 행동으로 옮길 점이 있으면 함께)

## (그 아래에 구체적 내용)
```

- takeaway는 **추상적 요약이 아니라 내 사업에 연결된 시사점**으로 쓴다.
- WELLNESS(여성향·일상·루틴) 신규 라인과 연결될 여지가 있으면 **반드시 짚는다**.

### 3-2. 페이지 표준 구조

```markdown
# 제목

> [!takeaway] 내 관점 핵심
> ...

## 핵심 요약
- 자료의 본질을 3~5줄 불릿으로.

## 상세
- 구체 내용. 필요시 소제목으로 구획.

## 내 사업 적용 (Action)
- 삼대오백 / MXSSIVE / WELLNESS 각각에 어떻게 쓸지.

## 관련 페이지
- [[연결된 페이지1]] — 왜 관련 있는지 한 줄
- [[연결된 페이지2]]

## 출처
- raw/파일명 또는 링크, 정리 날짜
```

(자료 성격에 따라 섹션은 가감 가능. 단 **takeaway callout과 관련 페이지(교차참조)는 항상 포함**.)

### 3-3. 교차참조 (적극적으로)

- 새 페이지를 만들거나 갱신할 때 **항상** 관련 기존 페이지를 찾아 `[[링크]]`를 건다.
- 가능하면 **양방향**: 기존 페이지에도 새 페이지로의 링크를 추가한다.
- 링크 옆에 **왜 연결되는지 한 줄 설명**을 붙인다.
- 파일명 = 위키링크 이름. 파일명은 `한글-케밥-형태.md` 권장 (공백 대신 하이픈).

### 3-4. 파일명 규칙

- 소문자/한글 + 하이픈, 공백 지양: `경쟁사-OO프로틴-분석.md`
- index의 카테고리 번호를 접두어로 쓰지 않는다 (분류는 index가 담당). 단 `00-비즈니스-맥락`만 예외(루트 맥락).

## 4. index.md 카테고리 (기본)

자료 유형에 맞춘 기본 카테고리. 필요시 Claude가 추가 제안.

1. **시장 · 경쟁** — 업계 동향 기사, 경쟁사 분석
2. **판매 · 마케팅 전략** — 판매전략, 부진재고 처리, 앰버서더, 유튜브/인스타, 오프라인 행사
3. **제품 · 법령** — 신제품 구상, 식품/광고 관련 법령 체크
4. **학습 · 노하우** — 강연/교육에서 배운 것
5. **프로젝트 · 기록** — 개인 프로젝트, 삽질(시행착오) 기록

각 카테고리 아래에 `- [[페이지명]] — 한 줄 설명` 형태로 나열한다.

## 5. 톤 & 언어

- 모든 위키 내용과 응답은 **한국어**로 작성한다.
- 군더더기 없이, 실무자가 바로 쓸 수 있게 구체적으로.
- 불확실하거나 위키에 근거 없는 내용은 추측이라고 명시한다.
